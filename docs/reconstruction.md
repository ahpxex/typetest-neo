# 打字测试系统重构计划

## 1. 文档目的

本文用于确定新一代打字测试系统的重构方向、技术选型、模块划分、数据模型、迁移策略、开发阶段和上线方案，供项目评审与后续实施使用。

本计划默认以下前提成立：

- 当前项目将基于现有的 Next.js 新项目继续开发。
- 系统主要服务对象为学校每年大一学生，总量通常小于 500 人。
- 系统部署方式以**单机/单实例自托管**为前提，而不是多实例 serverless。
- 业务以“稳定可用、便于维护、方便每年复用”为首要目标，而不是做成通用商业化打字平台。

---

## 2. 旧系统复盘结论

根据 `ref/typeeasy_final2022_10_18` 的旧代码分析，原系统核心业务并不复杂，主要能力包括：

### 2.1 学生侧

- 姓名 + 学号登录。
- 根据教师当前设置进入一篇指定文章。
- 进行限时打字测试。
- 前端实时显示：倒计时、速度、正确率、错误数、退格数、总字数。
- 提交成绩后查看排名。

### 2.2 教师/管理员侧

- 管理员账号登录。
- 设置当前测试文章：
  - 每天固定随机一篇。
  - 立即随机并锁定当前文章。
- 根据学号查询学生姓名。
- 手动新增学生。
- 导出指定时间段、本月、近几个月的成绩 Excel。

### 2.3 排名与记录

- 保存每次成绩记录。
- 支持今日排名、本班排名、本月排名、近 7 天排名。
- 记录学生姓名、学号、速度、正确率、测试时间、IP、文章来源、测试时长。

### 2.4 旧系统问题

旧系统能完成任务，但存在明显技术债：

- PHP + 旧式 `mysql_*` API，技术栈过时。
- 前端和后端耦合严重，HTML 页面大量静态复制。
- 计分逻辑主要在前端，服务端只接收结果，可信度不足。
- 权限模型非常薄弱，管理员账号硬编码。
- SQL 拼接多，输入校验与安全性不足。
- 导出、题库、学生名单、场次控制缺少明确的领域模型。
- 可维护性差，每年使用时都容易出现临时修补。

结论：**适合保留业务规则，不适合继续在旧代码上修补，应该以现代栈重写。**

## 3. 重构目标

### 3.1 一期目标（必须完成）

完成一套可替代旧系统的稳定版本，覆盖以下核心流程：

1. 管理员登录。
2. 学生名单导入与管理。
3. 文章题库管理。
4. 创建/配置当前测试场次。
5. 学生姓名 + 学号 + 校园邮箱登录测试。
6. 进入指定文章并限时打字。
7. 实时显示速度/正确率/进度。
8. 提交成绩并服务端校验。
9. 查看排行榜。
10. 导出成绩。

### 3.2 二期目标（可选增强）

- 支持多场次历史归档。
- 支持班级/专业/学院维度筛选。
- 支持中文文章模式。
- 支持异常成绩标记与人工复核。
- 支持学生个人历史记录查询。
- 支持更完善的后台统计报表。

### 3.3 非目标（当前阶段不做）

- 不做大型社交化打字社区。
- 不做实时对战。
- 不做复杂消息通知系统。
- 不做多校区 SaaS 多租户平台。
- 不以移动端为主战场，优先保证桌面端教室环境体验。

## 4. 技术决策

### 4.1 总体选型

- 前端框架：`Next.js`（App Router）
- 语言：`TypeScript`
- 样式：`Tailwind CSS`
- 数据库：`SQLite`
- ORM / 迁移：`Drizzle ORM` + `drizzle-kit`
- SQLite 驱动：`better-sqlite3`
- 表单校验：`zod`
- 测试：`Vitest` + `Playwright`
- 导出：优先 `CSV`，如确需原生 Excel 再补 `xlsx`
- 部署：单机 Node 服务，自托管

### 4.2 为什么选择 SQLite

选择 SQLite 的原因：

- 学生规模小，总数据量很有限。
- 提交成绩的写入密度不高，主要集中在测试结束时。
- 单机部署场景下，SQLite 能明显降低运维复杂度。
- 备份、迁移、恢复都简单。
- 不需要额外维护 MySQL / PostgreSQL 服务。

设计前提：

- 启用 `WAL` 模式。
- 所有写入都走短事务。
- 不在测试过程中按击键实时写库。
- 排行榜和管理查询以读为主。

不采用 SQLite 的场景：

- 若未来改为多实例部署。
- 若未来迁移到 Vercel 一类无共享本地文件系统的平台。
- 若未来需要跨地域或远程写库。

因此本计划明确：**当前版本按单实例自托管设计**。

### 4.3 为什么选择 Next.js + Tailwind

- 现项目已是 Next.js 新工程，可直接延续。
- App Router 适合把页面、服务端读取、写操作整合在一套项目里。
- 对后台管理页、学生测试页、排行榜页都足够成熟。
- Tailwind 能快速搭建一致的后台与考试界面。
- 后续如果要加中后台组件，也容易扩展。

### 4.4 为什么选择 Drizzle

- 与 TypeScript 配合自然。
- 对 SQLite 支持成熟。
- 迁移与 schema 定义直观。
- 相比重量级 ORM，更适合本项目的小而稳需求。
- 未来若切换到 libSQL / PostgreSQL，迁移成本相对可控。

### 4.5 身份认证策略

本项目不引入大型通用认证框架，采用**自定义轻量 session 方案**：

- 学生登录：姓名 + 学号 + 校园邮箱三项校验通过后，签发学生 session cookie。
- 管理员登录：用户名 + 密码，通过哈希校验后签发管理员 session cookie。
- session 存储方式：优先使用服务端数据库会话表 + HttpOnly cookie。

原因：

- 业务角色简单，只有学生和管理员。
- 不需要 OAuth、邮箱验证、第三方登录。
- 自定义会话更可控，也更便于校内自托管。

补充约束：

- 学生登录时，校园邮箱必须以 `@ucass.edu.cn` 结尾。
- 一期默认不做邮箱验证码链路，但要求 `students` 表中预存校园邮箱。
- 生产环境目标是按 `student_no + name + campus_email` 三项精确匹配，而不是只校验邮箱后缀。
- 开发环境允许直接使用预置测试学生账号和测试管理员账号，以便联调和回归测试。

## 5. 总体架构

系统采用单体 Web 应用架构，但在代码组织上按领域拆分。

### 5.1 架构原则

- 页面渲染尽量使用 Server Components。
- 数据写入用 Server Actions 或 Route Handlers。
- 交互密集的打字器、后台表格筛选等局部使用 Client Components。
- 计分规则抽成纯函数模块，前后端共享同一套核心算法。
- 所有权限校验在服务端落地，前端只做体验层限制。

### 5.2 逻辑分层

建议按以下层次拆分：

1. `app/`：路由、页面、布局、Route Handlers。
2. `features/`：按业务域拆分的 UI 与用例。
3. `lib/`：数据库、认证、权限、工具函数。
4. `db/`：schema、迁移、seed、查询封装。
5. `modules/typing/`：打字算法、文本处理、提交校验。
6. `components/`：跨页面通用组件。
7. `docs/`：设计文档与运维说明。

## 6. 业务模型设计

本次重构不再围绕“几张杂糅的表”来拼业务，而是先明确领域对象。

### 6.1 角色

- 学生：参加测试、查看个人结果与公共排行榜。
- 管理员：配置场次、管理题库、导入名单、查看与导出成绩。

### 6.2 核心领域对象

- 学生 `students`
- 班级/专业 `class_groups`
- 管理员 `admin_users`
- 文章 `articles`
- 测试场次 `campaigns`
- 场次文章分配 `campaign_articles`
- 测试记录 `attempts`
- 登录会话 `sessions`
- 操作审计 `audit_logs`（可选但推荐）

### 6.3 为什么要引入场次 `campaigns`

旧系统本质上已经有“当前测试文章”和“当前这次考试”的概念，只是没有建模清楚。

引入 `campaigns` 的好处：

- 每年的考试可以自然归档。
- 可以按场次控制文章、时间、开放状态。
- 可以区分练习和正式考试。
- 导出、排名、统计都能更精确。
- 未来支持“同一学年多次测试”时不用推翻结构。

## 7. 数据库表设计（一期）

以下为建议的核心表。字段以业务稳定和实现效率为优先，先避免过度复杂。

### 7.1 `class_groups`

用于表示班级、专业班、教学行政分组。

建议字段：

- `id`
- `grade_year`：年级，如 2026
- `department`
- `major`
- `name`：班级名称
- `code`：班级编码
- `created_at`
- `updated_at`

### 7.2 `students`

建议字段：

- `id`
- `student_no`：学号，唯一
- `name`
- `campus_email`：校园邮箱，唯一
- `class_group_id`
- `status`：`active` / `inactive`
- `notes`
- `created_at`
- `updated_at`

说明：

- 姓名不做唯一键。
- 学号做唯一约束。
- 校园邮箱做唯一约束，统一按小写保存。
- 校园邮箱必须使用 `@ucass.edu.cn` 域名。
- 后台支持导入与人工修正。

### 7.3 `admin_users`

建议字段：

- `id`
- `username`
- `password_hash`
- `display_name`
- `role`：一期可先只保留 `admin`
- `status`
- `last_login_at`
- `created_at`
- `updated_at`

### 7.4 `articles`

建议字段：

- `id`
- `title`
- `slug`
- `language`：`en` / `zh`
- `content_raw`
- `content_normalized`
- `char_count`
- `word_count`
- `difficulty_level`：可选
- `status`：`draft` / `published` / `archived`
- `source`
- `created_at`
- `updated_at`

说明：

- `content_normalized` 用于服务端计分，避免提交时再做重复预处理。
- 一期可以只做 `en`，但字段先保留多语言扩展位。

### 7.5 `campaigns`

表示一轮可进行的测试活动。

建议字段：

- `id`
- `name`
- `academic_year`
- `term`
- `mode`：`practice` / `exam`
- `status`：`draft` / `scheduled` / `active` / `closed` / `archived`
- `duration_seconds`
- `article_strategy`：`fixed` / `daily_random` / `shuffle_once`
- `start_at`
- `end_at`
- `allow_retry`
- `max_attempts_per_student`
- `ranking_visibility`：`public` / `class_only` / `hidden`
- `created_at`
- `updated_at`

### 7.6 `campaign_articles`

用于把文章和场次关联起来。

建议字段：

- `id`
- `campaign_id`
- `article_id`
- `sort_order`
- `is_active`
- `created_at`

说明：

- `fixed` 模式下，通常只有一篇启用文章。
- `daily_random` 和 `shuffle_once` 模式下，会从绑定文章集合中选出当前文章。

### 7.7 `campaign_current_article`

用于记录当前场次实际生效的文章选择结果。

建议字段：

- `id`
- `campaign_id`
- `article_id`
- `resolved_date`：用于 `daily_random`
- `resolved_by`
- `reason`：`manual` / `daily_random` / `shuffle_once`
- `created_at`

说明：

- 这张表可以避免每次进入测试时都重新计算并导致选择不一致。
- 也方便审计“当天到底给学生出的是哪篇”。

### 7.8 `attempts`

这是最核心的成绩表。

建议字段：

- `id`
- `campaign_id`
- `student_id`
- `article_id`
- `attempt_no`
- `status`：`started` / `submitted` / `expired` / `cancelled`
- `started_at`
- `submitted_at`
- `duration_seconds_allocated`
- `duration_seconds_used`
- `typed_text_raw`
- `typed_text_normalized`
- `char_count_typed`
- `char_count_correct`
- `char_count_error`
- `backspace_count`
- `paste_count`
- `suspicion_flags`：JSON 字段
- `wpm_or_kpm`
- `accuracy`
- `score_version`
- `client_meta`：JSON，存 UA、分辨率等
- `ip_address`
- `created_at`
- `updated_at`

说明：

- 一期只存最终提交文本和汇总统计，不存全量击键事件。
- `typed_text_raw` 用于复核，`typed_text_normalized` 用于稳定计分。
- 服务器端重新计算 `wpm_or_kpm` 和 `accuracy`，不直接信任前端结果。

### 7.9 `sessions`

建议统一存储学生和管理员会话。

建议字段：

- `id`
- `user_type`：`student` / `admin`
- `user_id`
- `token_hash`
- `expires_at`
- `created_at`
- `last_seen_at`
- `ip_address`
- `user_agent`

### 7.10 `audit_logs`（推荐）

建议字段：

- `id`
- `actor_type`
- `actor_id`
- `action`
- `target_type`
- `target_id`
- `payload`
- `created_at`

适用场景：

- 管理员修改场次。
- 手动切换文章。
- 导入学生名单。
- 删除或归档文章。

## 8. 关键业务流程设计

### 8.1 管理员配置流程

1. 管理员登录后台。
2. 导入或维护学生名单。
3. 创建测试场次。
4. 绑定文章集合。
5. 设定出题策略、时长、是否允许重试。
6. 发布场次。
7. 考试期间查看记录与导出。

### 8.2 学生考试流程

1. 学生输入姓名 + 学号 + 校园邮箱登录。
2. 服务端校验名单、校园邮箱域名和场次状态。
3. 服务端为该学生创建一条 `attempts` 记录，状态为 `started`。
4. 返回测试页面，附带：
   - 场次 ID
   - 尝试 ID
   - 文章内容
   - 服务器记录的开始时间
   - 倒计时总时长
5. 学生开始输入，前端实时计算展示指标。
6. 到时或手动提交时，将最终文本与前端统计附带提交。
7. 服务端依据文章和提交文本重新计算最终成绩。
8. 更新 `attempts` 为 `submitted`。
9. 跳转到结果页和排行榜页。

### 8.3 排行榜流程

排行榜应以**数据库中的服务端最终成绩**为唯一来源。

一期支持以下维度：

- 当前场次总榜
- 当前场次班级榜
- 当前场次学生个人最佳成绩
- 当前场次最近提交记录

如果确实需要复刻旧系统，也可补充：

- 今日榜
- 本月榜
- 最近 7 天榜

但建议把“场次榜”作为主模型，时间榜作为派生查询，而不是反过来。

## 9. 打字引擎设计

这是整个系统最重要的技术模块之一，必须单独设计。

### 9.1 设计原则

- 前端即时反馈快。
- 服务端最终结果可信。
- 计分规则单一、可测试、可版本化。

### 9.2 计分策略

一期建议采用简化且稳妥的规则：

- 测试文章按统一规则规范化。
- 学生提交文本也按同样规则规范化。
- 对规范化后的文本做逐字符比对。
- 统计：
  - 总输入字符数
  - 正确字符数
  - 错误字符数
  - 正确率
  - 速度（KPM 或 CPM）
- 速度以**正确字符数 / 实际用时**计算，而不是总输入字符数。

这样做的优点：

- 规则明确。
- 不容易被乱打字符刷高分。
- 易于做服务端复算。

### 9.3 建议保留的前端实时指标

- 倒计时
- 当前速度
- 当前正确率
- 已完成进度
- 错误数
- 退格数

### 9.4 建议由服务端最终认定的指标

- 最终速度
- 最终正确率
- 最终错误数
- 最终是否有效
- 是否存在异常标记

### 9.5 文本规范化规则

建议一期定义统一规范化函数：

- 统一换行风格。
- 合并连续空格或按策略保留单空格。
- 可选：忽略行尾多余空格。
- 英文场景统一处理中英文引号差异。
- 可选：大小写敏感与否由场次配置控制。

### 9.6 防作弊策略

客户端限制只是体验层，真正的可信性要靠服务端。

建议采用以下策略：

- 前端禁用粘贴，但服务端仍记录 `paste_count`，不信任客户端为 0。
- 服务端根据 `started_at` 和 `submitted_at` 计算实际用时。
- 若提交速度异常高，打上 `suspicion_flags`。
- 若完成时间远短于理论极限，标记可疑。
- 若前端上报统计与服务端复算差异过大，标记可疑。
- 后台允许按“正常 / 可疑 / 作废”筛选。

一期不必做太重的反作弊，但要做到“可识别、可审阅”。

### 9.7 算法实现原则

`modules/typing-engine/` 内建议至少包含：

- `normalize.ts`：文本标准化
- `diff.ts`：文本比对
- `score.ts`：速度、正确率、错误数计算
- `validation.ts`：结果合法性校验
- `score.test.ts`：算法单元测试

前端实时展示和服务端最终结算都调用同一套核心函数，避免双份规则漂移。

## 11. 权限与安全设计

### 11.1 基础原则

- 所有写操作必须在服务端做权限校验。
- 所有管理员页面必须验证管理员 session。

- 学号格式
- 校园邮箱格式与域名后缀
- 姓名长度
- 文章标题与内容
- 场次时间区间
- 导入文件格式

### 11.5 审计与追踪

建议记录：

- 管理员登录/退出
- 场次创建/发布/关闭
- 文章创建/编辑/删除

## 14. 迁移策略

迁移目标不是“把旧代码原样搬过来”，而是把**有价值的数据与规则迁移进新模型**。

### 14.1 需要迁移的内容

- 学生名单（含校园邮箱）
- 旧题库文章
- 历史成绩（如果学校有保留需求）
- 当前有效的测试规则

### 14.2 不迁移的内容

- 旧 PHP 页面结构
- 旧的 jQuery / 原始 JS 逻辑
- 旧后台样式与模板
- 旧系统隐含的临时调试逻辑

### 14.3 迁移步骤

### 第一步：清洗旧文章

来源：`ref/typeeasy_final2022_10_18/continue/*.html`

处理方式：

- 提取标题
- 提取正文
- 去除页面壳、广告、脚本
- 保存为结构化文章记录

### 第二步：导入学生名单

来源：旧 SQL 中 `student` 表。

处理方式：

- 导出为 CSV 或直接写一次性导入脚本
- 补充班级维度映射
- 补充校园邮箱字段
- 处理重复学号、异常姓名

注意：

- 旧系统的 `student` 表只有姓名和学号，没有校园邮箱。
- 因为新系统要求学生使用 `@ucass.edu.cn` 校园邮箱登录，所以邮箱数据不能直接从旧库完整迁移。
- 上线前需要从学校现有名册、教务系统或人工整理表中补齐 `campus_email` 后再导入。

### 第三步：迁移历史成绩（可选）

来源：旧 SQL 中 `user` 表。

处理方式：

- 建立到新 `attempts` 的映射
- 缺失字段如 `attempt_no`、`campaign_id` 用“legacy-import”策略补默认值
- 与新系统运行数据隔离展示

### 第四步：校验场次配置

旧系统中“当前文章”来自 `user_article` 表与后台切换逻辑。

新系统中应映射为：

- 一个当前 active 的 `campaign`
- 一组 `campaign_articles`
- 一条 `campaign_current_article` 结果

### 14.4 数据迁移脚本建议

建议单独维护以下脚本：

- `scripts/import-legacy-articles.ts`
- `scripts/import-legacy-students.ts`
- `scripts/import-legacy-attempts.ts`

要求：

- 可重复执行
- 支持 dry-run
- 输出导入统计与错误日志

## 19. 风险与应对

## 风险 1：SQLite 并发写入冲突

应对：

- 启用 WAL
- 短事务
- 提交只写一次结果
- 必要时设置 busy timeout

## 风险 2：前端实时分数与服务端最终分数不一致

应对：

- 使用共享核心算法
- 在结果页明确以服务端结算为准
- 对差异做日志记录

## 风险 3：旧数据质量差

应对：

- 编写迁移脚本时加清洗逻辑
- 导入前先 dry-run
- 将异常数据单独落日志

## 风险 4：管理员误操作切换场次/文章

应对：

- 关键动作加二次确认
- 加审计日志
- 允许回滚到上一个有效配置

## 风险 5：考试现场网络或浏览器异常

应对：

- 测试页尽量减少依赖复杂外部资源
- 提交前本地缓存当前输入内容
- 超时与刷新行为给出明确提示
