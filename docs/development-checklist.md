# 打字测试系统开发清单（当前版本）

本文基于当前已经明确的业务规则整理可实施的开发任务，并标出仍待确认的决策点。

## 1. 已确认的业务规则

- 学号是学生身份唯一主键。
- 学生登录表单保留：姓名、学号、校园邮箱。
- 校园邮箱由学号自动填充，默认规则为：`{studentNo}@ucass.edu.cn`。
- 学号结构：
  - 前 4 位：入学年份
  - 中间 5 位：专业代码
  - 中间 5 位的前 3 位：学院代码
  - 最后 2 位：班级/序号标识
- 不需要导入学生名单。
- 学生端需要区分“练习”和“考试”两条路径。
- 学生可无限登录，也允许考后继续练习。
- 管理员端需要支持按年份、学院、专业筛选学生。

## 2. 待确认的关键决策

### 2.1 邮箱认证形态

待和老师确认：

- 使用 magic link
- 使用邮箱验证码
- 或对接校内现有认证/发信体系

当前不要把系统绑定到第三方 SaaS 邮件服务（如 Resend）上。

### 2.2 发信基础设施

待确认可用渠道：

- 学校 SMTP
- 校内统一邮箱网关
- 教师可申请的发信账号
- 信息化部门提供的接口

开发时建议先把发信层抽象成 provider interface，避免和具体服务强耦合。

## 3. 推荐实施顺序

### P0：身份解析与认证基础

#### 3.1 新增学号解析模块

新增统一模块，例如 `lib/student-identity.ts`：

- `validateStudentNo(studentNo)`
- `parseStudentNo(studentNo)`
- `buildCampusEmail(studentNo)`
- `isCampusEmailMatch(studentNo, email)`

返回字段建议：

- `studentNo`
- `enrollmentYear`
- `schoolCode`
- `majorCode`
- `classSerial`
- `campusEmail`

#### 3.2 调整学生数据模型

`students` 表建议新增或调整字段：

- `studentNo`
- `name`
- `campusEmail`
- `enrollmentYear`
- `schoolCode`
- `majorCode`
- `status`
- `emailVerifiedAt`
- `lastLoginAt`
- `createdAt`
- `updatedAt`

要求：

- 首次认证成功后自动创建学生记录
- 后续登录自动更新姓名、邮箱、最近登录时间

#### 3.3 新增邮箱认证记录表

新增表，例如 `student_login_tokens`：

- `id`
- `studentNo`
- `campusEmail`
- `tokenHash` 或 `codeHash`
- `type`（`magic_link` / `code`）
- `expiresAt`
- `consumedAt`
- `createdAt`
- `requestIp`
- `requestUserAgent`

要求：

- 支持有效期控制
- 支持重复发送限频
- 支持单次消费
- 为审计和排错保留基础信息

#### 3.4 改造学生登录流程

学生登录改成两步：

第一步：填写身份信息
- 姓名
- 学号
- 校园邮箱（自动填充）

第二步：邮箱认证
- magic link 或验证码
- 成功后创建 session 并跳转学生首页

页面任务：

- 改造当前学生登录表单
- 学号输入后自动填充邮箱
- 提交时严格校验邮箱和学号一致
- 明确提示“系统通过校园邮箱认证”

#### 3.5 抽象发信服务层

新增统一接口，例如 `lib/mail/provider.ts`：

- `sendStudentLoginLink(...)`
- `sendStudentLoginCode(...)`

要求：

- 不直接写死第三方供应商
- 支持开发环境 fake provider（打印到日志）
- 支持未来接学校 SMTP 或内网服务

### P1：学生端练习 / 考试分流

#### 3.6 学生首页重做

登录成功后进入统一学生首页，而不是直接开考。

页面至少包含：

- 学生身份信息
- 开始练习
- 开始考试
- 最近练习记录
- 最近考试记录
- 个人最佳成绩摘要

建议路由：

- `/typing`：学生首页
- `/typing/practice`：练习模式
- `/typing/exam`：考试模式

#### 3.7 为成绩记录增加模式字段

`attempts` 表新增字段：

- `mode`：`practice | exam`

要求：

- 练习记录与考试记录都进入同一张表
- 排行榜只统计 `exam`
- 后台默认优先查看 `exam`
- 导出支持区分 `practice` 与 `exam`

#### 3.8 改造 typing 流程复用模式

复用现有打字页面逻辑，不做两套打字器。

需要改造的环节：

- 创建记录时带上 `mode`
- 提交记录时保留 `mode`
- 成绩页展示当前模式
- 学生历史页支持按模式分组

### P1：管理员后台改造

#### 3.9 删除无意义的名单导入逻辑

后台删除：

- 新增学生
- 批量导入 CSV

后台保留：

- 学生列表
- 搜索
- 启用/停用
- 查看成绩
- 导出数据

#### 3.10 增加年份 / 学院 / 专业筛选

后台学生列表增加筛选项：

- 年份
- 学院
- 专业

实现要求：

- 来源于 `students.enrollmentYear / schoolCode / majorCode`
- 与搜索框可叠加使用
- 支持导出当前筛选结果

#### 3.11 接入专业映射表

待拿到专业 `MAP` 后新增配置模块，例如：

- `lib/student-major-map.ts`

建议维护：

- `schoolCode -> schoolName`
- `majorCode -> majorName`

要求：

- 数据库存代码
- 页面和导出时映射为名称
- 缺失映射时优雅降级为代码显示

### P2：查询、导出与展示增强

#### 3.12 改造排行榜查询

要求：

- 只统计 `mode = exam`
- 可保留未来扩展空间，例如单独练习榜（当前非必需）

#### 3.13 改造后台详情弹窗

支持查看：

- 全部记录
- 仅练习记录
- 仅考试记录

要求：

- 继续保持避免重复请求/死循环的实现方式
- 展示模式、速度、正确率、用时、异常标记

#### 3.14 改造导出字段

导出建议包含：

- 学号
- 姓名
- 校园邮箱
- 入学年份
- 学院代码
- 学院名称
- 专业代码
- 专业名称
- 模式（练习/考试）
- 分数
- 正确率
- 用时
- 提交时间

## 4. 测试清单

### 4.1 认证链路

- 学号输入后邮箱自动填充
- 邮箱与学号不一致时不可提交
- 认证链接/验证码有效期正确
- 重复发送被限流
- 成功认证后自动创建或更新学生记录
- session 正常写入与失效

### 4.2 学生端

- 登录后进入学生首页
- 练习模式可正常开始、提交、查看记录
- 考试模式可正常开始、提交、进入正式成绩
- 练习成绩不进入正式排行榜
- 考试成绩进入正式排行榜
- 刷新、中断、重复提交不产生脏数据

### 4.3 管理员端

- 学生列表仅展示真实进入系统的学生
- 年份筛选正确
- 学院筛选正确
- 专业筛选正确
- 搜索与筛选组合正确
- 导出结果字段正确
- 查看成绩弹窗请求稳定

## 5. 当前最适合立刻开工的部分

即使认证发信方案尚未最终确定，以下部分也可以先做：

1. 学号解析模块
2. `students` 表结构扩展
3. `attempts.mode` 字段与练习/考试分流设计
4. 学生首页框架
5. 后台年份/学院/专业筛选数据结构
6. 专业映射模块接口设计
7. 发信 provider 抽象层（先不绑定具体供应商）

## 6. 当前不建议立即定死的部分

以下事项应等你和老师确认后再最终落地：

- magic link 还是验证码
- 具体发信服务或发信网关
- 邮件模板语气与学校名义
- 是否需要和校内其他认证系统联动
